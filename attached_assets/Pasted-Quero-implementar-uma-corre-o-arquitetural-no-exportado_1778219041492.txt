Quero implementar uma correção arquitetural no exportador PPTX v5.3.x para resolver o problema recorrente de destruição de tokens técnicos.

Contexto:
A arquitetura modular inspirada na Manus funcionou:
- planner por módulo;
- QA por módulo;
- merge global;
- fallback por módulo;
- redução de contaminação entre módulos.

Mas ainda persiste um problema central:
tokens técnicos são destruídos durante sanitização, reparo semântico, reflow ou QA.

Exemplos reais:
- "Criar estruturas condicionais com, e else." deveria ser "com if, elif e else."
- "Repetindo ações com e" deveria ser "com for e while."
- "Usar modos de abertura, e 'a'" deveria preservar "'r', 'w' e 'a'."
- "construtor init" deveria ser "__init__()".
- "Níveis como, e ERROR" deveria preservar "DEBUG, INFO e ERROR".
- "Organizar projetos Python com,." deveria preservar "src/ e tests/".
- "Gerenciar dependências com, e requirements.txt" deveria preservar "venv, pip e requirements.txt".

Diagnóstico:
O sistema está tentando reparar tokens depois que eles já foram danificados.
Isso gera uma espiral de regras e regex.

Nova estratégia:
Não reparar tokens técnicos depois.
Congelar/proteger tokens técnicos antes de qualquer sanitização/reparo.

Implementar uma camada de:

Technical Preservation Layer
Semantic Freezing
Token Masking
Re-hydration
Integrity Check

Inspirado na ideia:
texto técnico → congelar tokens → sanitizer/reparo só no texto natural → reidratar tokens → validar integridade.

NÃO reescrever o renderer.
NÃO migrar para v6.
NÃO alterar billing, schema ou rotas públicas.
NÃO adicionar novos layouts.
Implementar de forma incremental e segura.

1. Criar arquivo/módulo novo

Criar módulo separado, por exemplo:

supabase/functions/export-pptx-v4/technical-preservation.ts

Com funções principais:

- detectTechnicalTokens(text, context)
- protectTechnicalTokens(text, context)
- restoreTechnicalTokens(maskedText, tokenMap)
- validateTechnicalTokenIntegrity(originalText, restoredText, tokenMap)
- withTechnicalProtection(text, context, processorFn)

2. Conceito de Technical Token

Criar tipo:

TechnicalToken = {
  id: string;
  value: string;
  kind:
    | "python_keyword"
    | "python_builtin"
    | "python_dunder"
    | "python_method"
    | "python_exception"
    | "python_module"
    | "test_framework"
    | "log_level"
    | "file_path"
    | "filename"
    | "shell_command"
    | "package_tool"
    | "http_method"
    | "api_method"
    | "json_method"
    | "generic_identifier";
  start?: number;
  end?: number;
  protected: true;
}

3. Mascaramento

protectTechnicalTokens(text, context) deve substituir tokens técnicos por placeholders temporários neutros:

[[TECH_TOKEN_001]]
[[TECH_TOKEN_002]]
[[TECH_TOKEN_003]]

Retornar:

{
  maskedText,
  tokenMap,
  stats
}

Importante:
Os placeholders técnicos devem ser diferentes de placeholders de template e nunca aparecer no PPTX final.

4. Reidratação

restoreTechnicalTokens(maskedText, tokenMap) deve reinserir exatamente os valores originais.

Exemplo:

Original:
"Criar estruturas condicionais com if, elif e else."

Masked:
"Criar estruturas condicionais com [[TECH_TOKEN_001]], [[TECH_TOKEN_002]] e [[TECH_TOKEN_003]]."

Restored:
"Criar estruturas condicionais com if, elif e else."

6. Regex/context-aware detection

Detectar também padrões:

- palavras entre backticks: `...`
- trechos entre aspas simples que sejam modos/código: 'r', 'w', 'a', 'b', 'rb', 'wb'
- identificadores com parênteses: [a-zA-Z_][a-zA-Z0-9_]*()
- paths com /: src/, tests/, docs/
- arquivos com extensão: requirements.txt, setup.py, pyproject.toml
- nomes com underscore duplo: __init__()
- nomes com ponto: logging.basicConfig(), response.json()
- prefixo test_: test_soma, test_*

7. Onde aplicar no pipeline

Aplicar a proteção antes de qualquer operação que possa alterar texto:

- sanitizeTechnicalContent
- repairSemanticBreak
- repairLearningObjective
- repairTechnicalTokens
- detectBrokenSemanticSentence repair
- dedupe text normalization, se alterar texto final
- validate/rewrite slide title
- L1/L2/L3 cascade repairs
- any LLM rewrite/cascade repair

Fluxo padrão:

const protected = protectTechnicalTokens(text, context)
const processed = repairOrSanitize(protected.maskedText)
const restored = restoreTechnicalTokens(processed, protected.tokenMap)
validateTechnicalTokenIntegrity(text, restored, protected.tokenMap)

8. Criar helper comTechnicalProtection

Exemplo:

withTechnicalProtection(text, context, (masked) => {
  return repairSemanticBreak(masked, context)
})

Esse helper deve:
- mascarar tokens;
- executar processorFn;
- reidratar tokens;
- validar integridade;
- se reidratação falhar, rejeitar o reparo e manter o texto original ou bloquear.

9. Integrity Check

validateTechnicalTokenIntegrity deve verificar:

- todos os tokens do tokenMap foram reintroduzidos;
- nenhum placeholder [[TECH_TOKEN_...]] sobrou;
- nenhum token técnico original foi removido;
- nenhum token técnico foi duplicado indevidamente;
- nenhum token foi alterado parcialmente.

Se falhar:
- logar [TECH-PRESERVE-FAIL]
- rejeitar reparo;
- emitir QAIssue CRITICAL: TECHNICAL_TOKEN_LOSS
- fallback por módulo ou veto.

Adicionar TECHNICAL_TOKEN_LOSS ao HARD_CRITICAL_TYPES.
