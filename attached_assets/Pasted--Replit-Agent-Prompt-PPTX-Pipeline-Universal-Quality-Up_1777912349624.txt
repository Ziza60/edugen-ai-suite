# Replit Agent Prompt — PPTX Pipeline Universal Quality Upgrade (v5)

> **Objetivo**: Aplicar melhorias cirúrgicas no pipeline de geração de PPTX para elevar
> a qualidade visual e técnica de **todos** os decks futuros, independentemente de tema,
> idioma, paleta ou densidade. Nenhuma funcionalidade existente deve ser removida.

---

## Contexto do projeto

O arquivo principal é uma **Supabase Edge Function** em TypeScript (Deno) que:
1. Recebe `course_id`, `palette`, `theme`, `density`, `language` e `footerBrand` via POST
2. Busca módulos no banco Supabase
3. Chama **Gemini 2.5 Flash** para gerar JSON de slides por módulo
4. Usa **PptxGenJS 3.12** para renderizar o arquivo `.pptx`
5. Faz reparo do pacote ZIP com **JSZip** e sobe para Supabase Storage

Todas as alterações descritas abaixo devem ser aplicadas **nesse único arquivo**.
Se o agente identificar que partes do código foram refatoradas ou movidas, deve adaptar
as instruções ao estado atual do arquivo — o objetivo final importa mais que a localização exata.

## BLOCO 2 — Performance: paralelizar geração de slides

**Problema**: O loop sequencial aguarda cada chamada Gemini antes de iniciar a próxima.
Com 7 módulos isso resulta em ~21–28 segundos de espera linear desnecessária.

**Onde aplicar**: Dentro de `runPipeline()`, substituir o loop `for` de geração.

**Substituir** o loop de geração (o `for` que chama `generateModuleSlides`):
```typescript
// ANTES — sequencial
const allModuleSlides: Slide[][] = [];
for (let i = 0; i < modules.length; i++) {
  console.log(`[V4] Generating slides for module ${i + 1}/${modules.length}: "${modules[i].title}"`);
  const rawSlides = await generateModuleSlides(
    courseTitle, modules[i], i, density, language, geminiKey,
  );
  const splitSlides = splitOverflowSlides(rawSlides);
  const variedSlides = applyLayoutVariety(splitSlides);
  allModuleSlides.push(variedSlides);
}

// DEPOIS — paralelo com limite de concorrência (máximo 3 simultâneos)
async function processBatch<T>(
  items: T[],
  fn: (item: T, index: number) => Promise<Slide[]>,
  concurrency: number,
): Promise<Slide[][]> {
  const results: Slide[][] = new Array(items.length);
  let idx = 0;
  async function worker() {
    while (idx < items.length) {
      const i = idx++;
      results[i] = await fn(items[i], i);
    }
  }
  await Promise.all(Array.from({ length: concurrency }, worker));
  return results;
}

const allModuleSlides = await processBatch(
  modules,
  async (mod, i) => {
    console.log(`[V5] Generating module ${i + 1}/${modules.length}: "${mod.title}"`);
    const rawSlides = await generateModuleSlides(
      courseTitle, mod, i, density, language, geminiKey,
    );
    const splitSlides = splitOverflowSlides(rawSlides);
    const variedSlides = applyLayoutVariety(splitSlides);
    console.log(`[V5] Module ${i + 1}: ${rawSlides.length}→${variedSlides.length} slides`);
    return variedSlides;
  },
  3, // máximo 3 módulos em paralelo (respeita rate-limit Gemini)
);
```

**Atenção**: A função `processBatch` deve ser declarada **fora** de `runPipeline()`, no escopo
do módulo, para evitar re-declaração a cada execução.

```typescript
function buildPrompt(

  moduleTitle: string,
  moduleContent: string,
  moduleIndex: number,
  density: string,
  language: string,

  const maxItems = density === "compact" ? 4 : density === "detailed" ? 6 : 5;
  const maxItemChars = 90;
  const maxCodeLines = 12;

    .replace(/#{1,6}\s*/g, "")
    .replace(/\*{1,2}([^*]+)\*{1,2}/g, "$1")
    .replace(/[`_]/g, "")
    .replace(/\n{3,}/g, "\n\n")
    .trim()
    .slice(0, 3500);

  return `You are a world-class instructional designer specializing in premium slide decks.
Generate exactly ${nSlides} content slides for MODULE ${moduleIndex + 1} of a professional course.

COURSE: "${courseTitle}"
MODULE ${moduleIndex + 1}: "${moduleTitle}"

SOURCE CONTENT (use as primary reference — do not invent content not present here):
---
${contentSnippet}
---

═══ CRITICAL RULES (violations disqualify the response) ═══
1. ALL text must be in: ${language}
2. Slide titles: specific, descriptive, max 65 chars. NEVER use: "Introduction", "Overview",
   "Module ${moduleIndex + 1}", "Objectives", or a title identical to the module name.
3. Each bullet item: ONE concrete idea, max 15 words, max ${maxItemChars} chars.
   No numbering. No markdown. No escape sequences.
4. Max ${maxItems} items per slide.
5. No slide may have an empty items array (except layout: "comparison" which uses leftItems/rightItems).
6. VARIETY RULE: No more than 2 consecutive slides with the same layout type.
7. The LAST slide of the module must ALWAYS use layout: "takeaways".

═══ LAYOUT SELECTION GUIDE ═══

"bullets"    → Default. 3–5 concrete facts, definitions, or principles. Use when content

"cards"      → Exactly 3–4 named concepts. Format: "ConceptName: Brief explanation (≤12 words)".

"twocol"     → 6–8 short items that naturally split into two balanced groups (not random halves).

- takeaways items should start with an action verb or key insight noun
```

## Notas de compatibilidade

1. O campo `highlight` adicionado à interface `Design` é **opcional de usar** nos renderers
   existentes — apenas `buildDesign()` precisa populá-lo. Renderers futuros podem acessá-lo
   via `d.highlight` para criar elementos de destaque máximo (ex: badge "Conceito Chave").

2. A mudança de `Calibri` para `Cambria` no `titleFont` afeta **todos os slides de título**
   (covers, headers, module covers). Se o usuário preferir manter Calibri para um tema
   específico, pode ser sobreposto no futuro adicionando `titleFont` como parâmetro da request.

3. A paralelização com `processBatch` mantém a **ordem dos módulos** porque usa índice fixo
   (`results[i]`), não `push`. A ordem de renderização permanece idêntica ao código atual.
